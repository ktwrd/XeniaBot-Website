
New goodies for the Role Preservation system, and a big refactor to how logging is configured!

I told ya that I'd work more on Xenia now that I've cleaned up the data layer ;3

<h3 id="changes">Changes</h3>

- Added ability to "blacklist" roles in the Role Preserve module, which will prevent those roles from being given back to a user when they re-join. ([!59](https://github.com/ktwrd/XeniaBot/pull/59))
  - **NOTE:** This will not prevent other bots from giving roles back to users.
- Added auditing system for Role Preservation
- Added more logging for the Role Preservation module
- Refactored how log channels are configured ([!53](https://github.com/ktwrd/XeniaBot/pull/53))
- Add Server Log events for "Member Updated" ([!16](https://github.com/ktwrd/XeniaBot/issues/16)) and "Role Updated" ([!25](https://github.com/ktwrd/XeniaBot/issues/25))
  - **NOTE:** Server Log events for "Member Updated" is a bit spammy when mass granting roles, or when a user re-joins and role preservation is enabled. This will be fixed in a future update.
- Partially redesigned server moderation in the Web UI (<a href="#webui-comparison-moderationview">see comparison screenshots</a>)
- Fixed some stability issues when sending server log messages.
- Fixed random "Message Updated" events being sent for really old messages
  - For some reason Discord now includes the Embed URL in the message content for old messages, and those messages only get updated when someone looks at them. This could be a good way to see if a user is scraping your server for messages, or someone just looking back at the past. Either way, it's unnecessary strain on Xenia's infrastructure.
- Updated to Bootstrap v5.3 in the Web Panel ([!58](https://github.com/ktwrd/XeniaBot/pull/58))

<h4 id="serverlog-command-changes">Server Logging - Command Changes</h4>

Due to how the Server Logging was refactored, the commands that you'd use to add/remove a server log event have changed.

Specific server log "events" can now have multiple channels tied to them, instead of a specific event only being sent to a specific channel.

| Command                            | Usage |
| ---------------------------------- | ----- |
| `/log add-event <event> <channel>` | To start sending a specific event to a channel. "Fallback" will send all events that have or have not been configured. |
| `/log get-channel-events`          | List what events are configured for a channel |
| `/log get-channels`                | Get what channels are being used for a specific log event |
| `/log reset`                       | Reset configuration for Server Logging |
| `/log reset-channel`               | Remove all events from a specific channel |

Currently you cannot remove events from channels via slash commands. This is something you need to do in the Web UI.

<h4 id="logging-rolepreservation">Role Preservation - Logging</h4>

In previous versions of Xenia, some events will always send to a log channel for the "Member Join" event, specifically when Xenia is unable to give a role back to a user.

This behavior still exists in Xenia 1.17, but it will be overridden if you add a "Role Preserve" log event to a channel.

It's also probably a good idea if you look at <a href="/guide/about_rolepreserve" target="_blank">the updated role preservation guide</a>.

<h4 id="rolepreservation-blacklist">Role Preservation - Blacklist</h4>

In Xenia 1.17, you can now ignore specific roles from being given back to users when they re-join.

Currently, you can only add/remove blacklisted roles via slash commands, but in a future update you'll be able to do that in the Web UI.

For more information, take a look at the [Role Blacklist section of the Role Preserve guide](/guide/about_rolepreserve#content-role-blacklist).
